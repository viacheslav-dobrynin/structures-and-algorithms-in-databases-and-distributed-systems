# R-tree

## Мотивация

Классические индексные структуры, такие как B-дерево, эффективно работают с одномерными данными — числами, строками, датами. Однако при работе с пространственными данными (координаты на плоскости, географические объекты, прямоугольники) возникает фундаментальная проблема: B-дерево способно упорядочить данные лишь по одному измерению.

Рассмотрим задачу: «найти все рестораны в радиусе 500 метров от заданной точки». Если хранить координаты в двух отдельных столбцах (широта и долгота), можно создать B-tree индексы по каждому из них, но эффективно использовать оба индекса одновременно невозможно. Составной B-tree индекс `(x, y)` помогает лишь при точном совпадении по первому компоненту, а при диапазонном поиске по обоим координатам он деградирует до последовательного сканирования по второму измерению.

Для решения этой задачи Антонин Гуттман (Antonin Guttman) в 1984 году предложил структуру данных **R-дерево** (R-tree), где «R» означает «rectangle» (прямоугольник). R-дерево организует пространственные объекты иерархически, группируя близко расположенные объекты и оборачивая каждую группу минимальным ограничивающим прямоугольником.

## Структура R-дерева

### Минимальный ограничивающий прямоугольник (MBR)

Центральным понятием R-дерева является **MBR** (Minimum Bounding Rectangle) — наименьший прямоугольник, стороны которого параллельны осям координат и который полностью содержит заданный пространственный объект или группу объектов.

Для двумерного случая MBR определяется четырьмя значениями: `(x_min, y_min, x_max, y_max)`.

Любой пространственный объект — точка, линия, полигон — может быть аппроксимирован своим MBR. Это позволяет свести сложные геометрические проверки к простым сравнениям координат прямоугольников.

### Узлы и свойства дерева

R-дерево является сбалансированным деревом поиска со следующими свойствами:

- **Листовые узлы** содержат записи вида `(MBR, указатель на объект)`, где MBR — ограничивающий прямоугольник самого пространственного объекта.
- **Внутренние узлы** содержат записи вида `(MBR, указатель на дочерний узел)`, где MBR — ограничивающий прямоугольник, покрывающий все MBR дочернего узла.
- Каждый узел (кроме корня) содержит от `m` до `M` записей, где `M` — максимальная ёмкость узла, а `m <= M/2` — минимальное заполнение.
- Корневой узел содержит от 2 до `M` записей (если дерево не состоит из одного узла).
- Все листья находятся на одном уровне — дерево сбалансировано.

Типичная структура R-дерева:

```
           [R0]                     <- корень (MBR покрывает всё)
          /    \
      [R1]      [R2]               <- внутренние узлы
      / \        / \
   [L1] [L2]  [L3] [L4]           <- листовые узлы
   /|\   /|\   /|\   /|\
  объекты...                       <- пространственные объекты
```

Параметр `M` обычно выбирается исходя из размера дисковой страницы. Например, при размере страницы 4 КБ и размере записи 36 байт (четыре координаты по 8 байт + указатель 4 байта) в один узел помещается около 100 записей.

Важное свойство R-дерева, отличающее его от B-дерева: **MBR соседних узлов на одном уровне могут перекрываться**. Это означает, что при поиске может потребоваться обход нескольких ветвей дерева.

## Операции

### Поиск (Search)

#### Поиск в диапазоне (Range Query)

Задача: найти все объекты, пересекающиеся с заданной областью поиска `S`.

Алгоритм начинается с корневого узла и рекурсивно обходит дерево:

1. Если текущий узел — **внутренний**, проверить каждую запись `(MBR_i, child_i)`. Если `MBR_i` пересекается с `S`, рекурсивно выполнить поиск в `child_i`.
2. Если текущий узел — **листовой**, проверить каждую запись `(MBR_i, obj_i)`. Если `MBR_i` пересекается с `S`, добавить `obj_i` в результат. При необходимости выполнить точную геометрическую проверку (поскольку MBR является лишь аппроксимацией объекта).

```
Функция Search(узел N, область S):
    если N — листовой узел:
        для каждой записи E в N:
            если E.MBR пересекается с S:
                добавить E.объект в результат
    иначе:
        для каждой записи E в N:
            если E.MBR пересекается с S:
                Search(E.дочерний_узел, S)
```

Поиск использует двухэтапную фильтрацию:
1. **Этап фильтрации** — отсечение заведомо не подходящих поддеревьев по MBR.
2. **Этап уточнения** — точная геометрическая проверка объектов-кандидатов.

#### Точечный запрос (Point Query)

Частный случай поиска в диапазоне, где область поиска — это точка. Алгоритм аналогичен, но проверяется, содержится ли точка внутри MBR.

### Вставка (Insert)

Вставка нового объекта в R-дерево происходит в несколько этапов.

#### Алгоритм ChooseLeaf

Цель — выбрать листовой узел, наиболее подходящий для размещения нового объекта.

1. Начать с корня.
2. Если текущий узел — листовой, вернуть его.
3. Иначе выбрать ту дочернюю запись, MBR которой потребует **наименьшего увеличения площади** для включения MBR нового объекта. При равенстве выбрать запись с меньшей площадью MBR.
4. Перейти к выбранному дочернему узлу и повторить с шага 2.

```
Функция ChooseLeaf(корень, новый_MBR):
    N = корень
    пока N не листовой:
        выбрать запись E в N, для которой
            увеличение площади E.MBR при включении новый_MBR минимально
        N = E.дочерний_узел
    вернуть N
```

#### Вставка и разделение узла

После выбора листового узла:

1. Если в узле есть свободное место (`|записей| < M`), добавить новую запись и обновить MBR по пути от листа к корню.
2. Если узел заполнен, выполнить **разделение узла** (split).

При разделении `M + 1` записей необходимо распределить по двум узлам так, чтобы:
- каждый узел содержал не менее `m` записей;
- суммарная площадь MBR двух новых узлов была минимальной.

Точное решение задачи оптимального разделения имеет экспоненциальную сложность, поэтому используются эвристические стратегии.

**Линейное разделение (Linear Split)** — сложность `O(M)`:

1. Для каждого измерения найти пару записей с наибольшим расстоянием между их MBR (нормализованным по ширине всех записей вдоль этого измерения). Выбрать пару с наибольшим нормализованным расстоянием — это «семена» (seeds) двух новых групп.
2. Оставшиеся записи распределить последовательно: каждую запись добавить в ту группу, MBR которой увеличится меньше.

**Квадратичное разделение (Quadratic Split)** — сложность `O(M^2)`:

1. **PickSeeds**: для каждой пары записей `(E_i, E_j)` вычислить «потерю» — площадь MBR, покрывающего обе записи, минус площади их индивидуальных MBR. Выбрать пару с максимальной потерей.
2. **PickNext**: из оставшихся записей выбрать ту, для которой разница в увеличении площади MBR между двумя группами максимальна. Добавить её в группу, требующую меньшего увеличения.
3. Повторять шаг 2, пока все записи не будут распределены. Если одна из групп должна получить все оставшиеся записи для выполнения условия `m`, назначить их в эту группу.

После разделения листового узла аналогичное разделение может потребоваться и для родительского узла (каскадное разделение вверх по дереву, аналогично B-дереву). Если разделяется корень, создаётся новый корень с двумя дочерними узлами, и высота дерева увеличивается на 1.

### Удаление (Delete)

Удаление объекта из R-дерева отличается от аналогичной операции в B-дереве: вместо заимствования записей у соседних узлов используется механизм повторной вставки.

#### Алгоритм

1. Найти листовой узел `L`, содержащий удаляемую запись.
2. Удалить запись из `L`.
3. Вызвать **CondenseTree**, начиная с `L`.

#### Алгоритм CondenseTree

1. Пусть `N = L`. Создать пустой список `Q` для «осиротевших» записей.
2. Если `N` — корень, перейти к шагу 5.
3. Пусть `P` — родитель `N`. Если `N` содержит менее `m` записей:
   - Удалить запись для `N` из `P`.
   - Добавить все записи из `N` в `Q`.
4. Если `N` содержит не менее `m` записей, обновить MBR в `P`.
5. Присвоить `N = P` и перейти к шагу 2.
6. Повторно вставить все записи из `Q` в дерево с помощью операции Insert.

```
Функция CondenseTree(узел N):
    Q = []
    пока N не корень:
        P = родитель(N)
        если |N.записи| < m:
            удалить запись N из P
            Q += N.записи
        иначе:
            обновить MBR для N в P
        N = P
    для каждой записи E в Q:
        Insert(E)
```

Повторная вставка гарантирует, что осиротевшие записи будут размещены в наиболее подходящих узлах, что может улучшить общую структуру дерева.

Если после CondenseTree корень содержит только одну дочернюю запись (и корень не является листом), этот единственный дочерний узел становится новым корнем, и высота дерева уменьшается на 1.

## Варианты и улучшения

### R*-дерево

R*-дерево (Beckmann et al., 1990) — наиболее распространённый на практике вариант, включающий несколько ключевых улучшений:

- **Улучшенный выбор поддерева (ChooseSubtree)**: для узлов, указывающих на листья, вместо минимизации увеличения площади минимизируется увеличение **перекрытия** (overlap) между MBR. Это уменьшает количество путей, которые необходимо обойти при поиске.
- **Улучшенное разделение узла**: при разделении рассматриваются все возможные разбиения вдоль каждой оси (записи сортируются по каждому измерению). Выбирается ось с наименьшей суммарной длиной периметра, затем — разбиение с минимальным перекрытием.
- **Принудительная повторная вставка (Forced Reinsertion)**: при переполнении узла вместо немедленного разделения часть записей (обычно 30%) удаляется из узла и вставляется заново. Это позволяет перераспределить записи между узлами и уменьшить перекрытие MBR, что может предотвратить разделение.

R*-дерево обеспечивает значительно лучшую производительность поиска по сравнению с оригинальным R-деревом благодаря уменьшению перекрытий между узлами.

### R+-дерево

R+-дерево (Sellis et al., 1987) решает проблему перекрытия MBR радикальным образом: **MBR узлов одного уровня не пересекаются**. Это достигается за счёт того, что объект, пересекающий несколько MBR, **дублируется** в нескольких узлах.

Преимущества:
- Точечный запрос всегда проходит ровно по одному пути от корня до листа.
- Поиск в диапазоне более предсказуем.

Недостатки:
- Объекты могут храниться в нескольких листовых узлах, что увеличивает потребление памяти.
- Вставки и удаления сложнее из-за необходимости поддерживать согласованность дубликатов.

## Сложность

| Операция           | Среднее время       | Худшее время       |
|--------------------|--------------------|-------------------|
| Поиск              | `O(log_M n)`       | `O(n)`            |
| Вставка            | `O(log_M n)`       | `O(n)`            |
| Удаление           | `O(log_M n)`       | `O(n)`            |
| Пространство       | `O(n)`             | `O(n)`            |

Здесь `n` — количество объектов, `M` — максимальная ёмкость узла.

В среднем случае высота дерева равна `O(log_M n)`, и на каждом уровне просматривается ограниченное число узлов. Однако при сильном перекрытии MBR (что характерно для неравномерно распределённых данных) может потребоваться обход значительной части дерева, что приводит к линейной сложности в худшем случае.

На практике при использовании R*-дерева перекрытия минимизированы, и производительность поиска близка к `O(log_M n)` для большинства реальных наборов данных.

## Применение в базах данных

### R-дерево в PostgreSQL и PostGIS

PostgreSQL реализует R-дерево не как отдельный тип индекса, а через механизм **GiST** (Generalized Search Tree) — обобщённого дерева поиска. GiST предоставляет фреймворк для реализации различных древовидных индексов, и R-дерево является одной из его конкретных реализаций.

При создании пространственного индекса в PostGIS:

```sql
CREATE INDEX idx_geom ON my_table USING GIST (geom);
```

PostgreSQL создаёт GiST-индекс, который для пространственных типов данных (geometry, geography) работает по принципу R-дерева: внутренние узлы хранят MBR, а поиск использует двухэтапную фильтрацию.

Оператор `&&` в PostGIS проверяет пересечение MBR двух геометрий и используется на этапе фильтрации:

```sql
SELECT * FROM my_table WHERE geom && ST_MakeEnvelope(0, 0, 10, 10);
```

Функции вроде `ST_Contains`, `ST_Intersects`, `ST_DWithin` внутренне используют оператор `&&` для предварительной фильтрации по индексу, а затем выполняют точную геометрическую проверку.

Подробнее о практическом использовании пространственных индексов в PostGIS см. раздел [PostGIS](./postgis.mdx).
