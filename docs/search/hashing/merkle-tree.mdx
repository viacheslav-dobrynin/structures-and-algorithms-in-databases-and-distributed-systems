# Дерево Меркла

## Мотивация

Криптографическая хеш-функция `H` сжимает произвольный набор данных в строку фиксированной длины `k` бит и обладает свойствами collision-resistance, second-preimage и preimage. Этого достаточно, чтобы убедиться в целостности **одного** объекта: сторона-проверяющая хранит `H(data)` и сравнивает его с пересчитанным значением.

Однако в реальных системах объект чаще всего составной: блок транзакций, дерево файлов в репозитории, набор сегментов реплицируемой таблицы. Для таких сценариев одной плоской хеш-функции мало — она порождает три практические проблемы:

- **Целостность с локальностью.** Если хранить «один большой хеш» от всего набора, любая модификация одного элемента инвалидирует весь хеш и не позволяет указать, **что именно** изменилось. Это допустимо для скачанного ISO-образа, но непригодно для версионируемого репозитория или реплицируемой БД.
- **Inclusion-проверка без полной выгрузки.** Лёгкому клиенту (light client, SPV-кошелёк, CT-аудитор) нужно убедиться, что конкретный элемент `data_i` входит в состав опубликованного агрегата, **не скачивая** весь набор `{data_1, ..., data_n}`. Простой список чексумм `[H(data_1), ..., H(data_n)]` решает эту задачу, но требует `O(n)` доверенных байт.
- **Anti-entropy между репликами.** При сравнении двух копий распределённой структуры нужно быстро локализовать различающиеся подмножества. Поэлементное сравнение даёт `O(n)` обменов; хочется `O(diff · log n)`.

Бинарное **дерево Меркла**, предложенное Ральфом Мерклом в его докторской диссертации 1979 года [[1](#ref-merkle-1979)] и развитое в работе CRYPTO '87 [[2](#ref-merkle-1987)], решает все три задачи: один корневой хеш фиксированного размера авторизует весь набор, а доказательство включения отдельного элемента имеет размер `O(log n) · k` бит.

Эта конструкция стала стандартом де-факто в системах, где целостность нужно доказывать выборочно или сравнивать большие наборы:

- **Bitcoin** — `merkle root` транзакций в заголовке блока [[5](#ref-nakamoto)];
- **Ethereum** — три корня (`stateRoot`, `transactionsRoot`, `receiptsRoot`) на основе Merkle Patricia Trie [[6](#ref-yellowpaper)];
- **Git** — Merkle DAG из коммитов, деревьев и блобов [[7](#ref-git-internals)];
- **Certificate Transparency** — публичные append-only логи сертификатов согласно RFC 6962 [[3](#ref-rfc-6962)];
- **IPFS** — content-addressable хранилище на Merkle DAG [[9](#ref-ipfs)];
- **ZFS, Btrfs, dm-verity** — проверяемое чтение и verified boot;
- **Apache Cassandra, Amazon DynamoDB** — anti-entropy при репликации [[8](#ref-dynamo)] [[13](#ref-cassandra-repair)].

## Определение и структура

**Дерево Меркла** — это полное (или почти полное) бинарное дерево, в котором каждый лист содержит хеш одного блока данных, а каждый внутренний узел — хеш конкатенации хешей своих детей. Корень такого дерева (Merkle root) криптографически привязывает весь набор листьев к одному значению фиксированной длины.

Формально, пусть `H : {0,1}^* → {0,1}^k` — криптографическая хеш-функция (например, SHA-256 даёт `k = 256`, BLAKE3 — `k = 256`, Keccak-256 — `k = 256`). Для упорядоченного набора блоков данных `D = (d_1, ..., d_n)`:

- **листья**: `L_i = H(d_i)` для `i = 1, ..., n`;
- **внутренние узлы**: `N(left, right) = H(left || right)`, где `||` — конкатенация байт;
- **корень**: `root` — единственный узел без родителя, агрегирующий все листья.

В стойких к second-preimage реализациях вводят **доменное разделение** — отдельные префиксы для листа и для внутреннего узла. Например, RFC 6962 §2.1 [[3](#ref-rfc-6962)] фиксирует:

```
MTH({d}) = SHA-256(0x00 || d)                               (лист)
MTH(D[n]) = SHA-256(0x01 || MTH(D[0:k]) || MTH(D[k:n]))     (внутренний узел)
```

где `k` — наибольшая степень двойки, строго меньшая `n`. Без префиксов внутренний узел и лист различаются только по семантике и могут совпасть по биту, что открывает атаки на коллизии (см. [Безопасность](#безопасность-атаки-и-контрмеры)).

### Листья и внутренние узлы

Дерево над `n` листьями имеет высоту `h = ⌈log_2 n⌉`. Если `n` — точная степень двойки, дерево полностью сбалансировано; иначе правое поддерево короче, форма однозначно определяется числом `n` (RFC 6962, §2.1).

### Схема: 8 листьев

Бинарное дерево Меркла над восемью блоками данных схематически выглядит так (служебные префиксы `0x00`/`0x01` опущены для краткости):

<pre>{`                       root = H(N12 || N34)
                       /                    \\
              N12 = H(A || B)        N34 = H(C || D)
              /        \\              /        \\
        A = H(L1||L2)  B = H(L3||L4)  C = H(L5||L6)  D = H(L7||L8)
         /    \\         /    \\         /    \\         /    \\
        L1    L2       L3    L4       L5    L6       L7    L8
        |     |        |     |        |     |        |     |
       d1    d2       d3    d4       d5    d6       d7    d8`}</pre>

Здесь `L_i = H(d_i)` — листья, `A, B, C, D, N12, N34, root` — внутренние узлы. Корень `root` — это `k`-битное значение, фиксирующее все восемь блоков `d_1, ..., d_8`.

### Числовой пример

Возьмём четыре блока `d_1, d_2, d_3, d_4` и обозначим хеши символическими плейсхолдерами (значения **иллюстративны**, не результаты реального вызова SHA-256):

| Шаг | Узел | Формула | Иллюстративное значение |
|---|---|---|---|
| 1 | `H_1` | `H(d_1)` | `0xA1...` |
| 1 | `H_2` | `H(d_2)` | `0xB2...` |
| 1 | `H_3` | `H(d_3)` | `0xC3...` |
| 1 | `H_4` | `H(d_4)` | `0xD4...` |
| 2 | `H_{12}` | `H(H_1 \|\| H_2)` | `0xE5...` |
| 2 | `H_{34}` | `H(H_3 \|\| H_4)` | `0xF6...` |
| 3 | `root` | `H(H_{12} \|\| H_{34})` | `0x07...` |

Корень `root` — единственное `k`-битное значение, которое нужно опубликовать или подписать: его достаточно, чтобы любой проверяющий мог удостовериться в неизменности любого из четырёх блоков, имея сам блок и `O(log n)` промежуточных хешей.

## Свойства и инварианты

Из определения дерева Меркла прямо следуют его ключевые свойства:

- **Tamper-evidence.** Любая модификация хотя бы одного бита в любом листе `d_i` пересчитывает все хеши на пути от листа до корня. Стойкость хеш-функции к коллизиям гарантирует, что злоумышленник не может подобрать другую последовательность блоков с тем же `root`.
- **Локальность audit path.** Для проверки включения блока `d_i` достаточно `⌈log_2 n⌉` хешей-сиблингов вдоль пути от листа до корня — независимо от того, насколько велико `n`. Например, для `n = 2^{20}` audit path содержит ровно 20 хешей.
- **Размер доказательства.** Merkle proof для одного листа: `⌈log_2 n⌉ · k` бит. При SHA-256 (`k = 256`) и `n = 2^{30}` это `30 · 32 = 960` байт.
- **Чувствительность к порядку листьев.** Дерево фиксирует не только содержимое, но и **порядок** блоков: перестановка двух транзакций в блоке Bitcoin даёт другой `root`. Этот инвариант критичен для многих применений (Bitcoin, CT) и нежелателен для других (anti-entropy между двумя репликами с одним и тем же набором, но разной локальной сортировкой).
- **Caveat: безопасность не выше хеш-функции.** Если `H` теряет collision-resistance (как SHA-1 после атаки SHAttered, 2017), теряет защиту и всё дерево. Поэтому современные реализации мигрируют на SHA-256, BLAKE3 или Keccak-256.

## Доказательства включения (Merkle proof)

**Audit path** (или **Merkle proof**) — это упорядоченный список хешей-сиблингов на пути от листа `L_i` к корню. Имея сам блок `d_i`, audit path и опубликованный `root`, проверяющий пересчитывает корень снизу вверх и сравнивает с известным значением.

### Псевдокод: построение audit path

```python
def build_audit_path(leaves, index):
    """
    leaves — список листовых хешей (порядок фиксирован);
    index — позиция целевого листа (0-индексация);
    возвращает список (sibling_hash, side), где side ∈ {'L', 'R'}.
    """
    path = []
    nodes = list(leaves)
    while len(nodes) > 1:
        # выровнять до чётного числа узлов согласно правилу RFC 6962
        if index ^ 1 < len(nodes):
            sibling = nodes[index ^ 1]
            side = 'R' if index % 2 == 0 else 'L'
            path.append((sibling, side))
        # подняться на уровень выше
        next_level = []
        for i in range(0, len(nodes) - 1, 2):
            next_level.append(H(b'\x01' + nodes[i] + nodes[i + 1]))
        if len(nodes) % 2 == 1:
            next_level.append(nodes[-1])  # перенос «лишнего» хеша на уровень выше
        nodes = next_level
        index //= 2
    return path
```

### Псевдокод: проверка audit path

```python
def verify(leaf_data, audit_path, expected_root):
    h = H(b'\x00' + leaf_data)               # доменный префикс листа
    for sibling, side in audit_path:
        if side == 'R':
            h = H(b'\x01' + h + sibling)     # сиблинг справа
        else:
            h = H(b'\x01' + sibling + h)     # сиблинг слева
    return h == expected_root
```

### Пример: audit path для `d_3`

Для дерева из четырёх листьев (см. [Числовой пример](#числовой-пример)) audit path для `d_3` состоит из двух хешей:

- `H_4` со стороной `R` (правый сиблинг `d_3` на уровне листьев);
- `H_{12}` со стороной `L` (левый сиблинг `H_{34}` на уровне внутренних узлов).

Проверка пересчитывает: `h := H_3 = H(d_3) → H(H_3 || H_4) = H_{34} → H(H_{12} || H_{34}) = root`. Если итоговое значение совпадает с опубликованным `root`, включение доказано.

### Сложность доказательства

| Параметр | Значение |
|---|---|
| Размер audit path | `⌈log_2 n⌉` хешей |
| Размер в битах | `⌈log_2 n⌉ · k` |
| Время верификации | `⌈log_2 n⌉` хеш-вычислений |

Для `k = 256` бит (SHA-256) и `n = 2^{30}`: 30 хешей, 960 байт, 30 вызовов `H` на проверку — на любом современном CPU единицы микросекунд.

## Алгоритмы построения и обновления

### Построение снизу вверх

Каноническое построение — bottom-up: сначала вычисляются все листовые хеши, затем уровень за уровнем — внутренние узлы. Время `O(n)`, память `O(n)` (или `O(log n)`, если узлы выбрасываются после использования и нужен только `root`).

```python
def build_root(blocks):
    level = [H(b'\x00' + b) for b in blocks]              # листья
    while len(level) > 1:
        nxt = []
        for i in range(0, len(level) - 1, 2):
            nxt.append(H(b'\x01' + level[i] + level[i + 1]))
        if len(level) % 2 == 1:
            nxt.append(level[-1])                          # split-rule RFC 6962
        level = nxt
    return level[0]
```

### Нечётное число листьев: Bitcoin vs RFC 6962

Когда `n` не является степенью двойки, нужно правило для «лишнего» листа. Существуют два разных подхода:

- **Bitcoin.** Если на каком-то уровне число узлов нечётно, последний узел **дублируется** и хешируется сам с собой [[5](#ref-nakamoto)]. Эта схема страдает от уязвимости **CVE-2012-2459** [[12](#ref-cve-2012-2459)]: злоумышленник может подобрать дублирующую транзакцию так, чтобы дерево с разным числом листьев давало один и тот же корень — вектор отказа в обслуживании. Эксплуатация была закрыта на уровне валидации Bitcoin Core, но дизайн-уязвимость в самой схеме осталась как поучительный пример.
- **RFC 6962, §2.1** [[3](#ref-rfc-6962)]. Дерево разбивается асимметрично: `k` — наибольшая степень двойки, строго меньшая `n` (`k < n ≤ 2k`); левое поддерево содержит первые `k` листьев и полностью сбалансировано, правое — оставшиеся `n − k` и строится рекурсивно. Никакого дублирования. Форма дерева однозначно определяется числом `n`.

Современные системы (Certificate Transparency, IPFS, etcd snapshot trees) используют RFC-6962-подобные правила. Дублирующее правило Bitcoin сохраняется только ради обратной совместимости с протоколом.

### Обновление и append

- **Изменение одного листа.** Пересчёт хешей только на пути от изменённого листа до корня — `O(log n)` хеш-вычислений, при условии, что промежуточные хеши кэшированы.
- **Добавление листа (append-only лог CT).** В среднем `O(log n)` амортизированно: реализация ведёт стек «правых» хешей по аналогии с двоичным счётчиком; при удвоении уровня высота растёт на единицу.

## Безопасность: атаки и контрмеры

Дерево Меркла наследует security-параметры от использованной хеш-функции, но добавляет собственные подводные камни.

- **Second-preimage / domain confusion.** Если внутренние узлы и листья хешируются одной функцией без префикса, у атакующего появляется свобода: подменив сам блок данных хешем-конкатенацией двух листьев, он может построить «параллельное» поддерево с тем же корнем. RFC 6962 §2.1 [[3](#ref-rfc-6962)] предписывает обязательные префиксы `0x00` для листьев и `0x01` для внутренних узлов; это лечит уязвимость на уровне дизайна.
- **CVE-2012-2459 (Bitcoin)** [[12](#ref-cve-2012-2459)]. Дублирование «лишнего» листа на нечётных уровнях позволяло разным наборам транзакций давать один и тот же merkle root — вектор block-malleability и DoS. Bitcoin Core закрыл уязвимость проверкой на уровне валидатора, но сам факт показывает, что split-rule из RFC 6962 строже и предпочтительнее.
- **Слабая хеш-функция.** SHA-1 после атаки **SHAttered** (2017) [[14](#ref-shattered)] считается небезопасной для целей коллизионной стойкости — Git с тех пор переходит на SHA-256 (`extensions.objectFormat = sha256`). Для новых систем рекомендуются SHA-256, SHA-3 / Keccak-256 или BLAKE3. Постквантовая угроза — алгоритм Гровера снижает security level хеш-функции с `k` до `k/2` бит, что для SHA-256 даёт ~128-битную стойкость к коллизиям и считается достаточным запасом.
- **Утечка структуры через timing.** В обычных реализациях верификации стороннего канала нет, но при работе с приватными данными (zk-proof'ы) важно проверять audit path в постоянное время.

## Сложность

Все оценки даны в худшем случае; `n` — число листьев, `k` — длина хеша в битах.

| Операция | Время | Память | Размер артефакта |
|---|---|---|---|
| Построение `root` | `O(n)` | `O(n)` или `O(log n)` потоково | `k` бит |
| Audit path для одного листа | `O(log n)` | `O(log n)` | `⌈log_2 n⌉ · k` бит |
| Верификация audit path | `O(log n)` | `O(1)` | — |
| Обновление одного листа (с кэшем) | `O(log n)` | `O(n)` для кэша всех узлов | — |
| Append (амортизированно, append-only лог) | `O(log n)` | `O(log n)` для side-stack | — |

## Применения

### Git

Внутренний объектный граф Git — каноничный пример **Merkle DAG**. Объекты четырёх типов (`blob`, `tree`, `commit`, `tag`) идентифицируются хешем своего сериализованного содержимого, и каждый объект ссылается на нижележащие через их хеши [[7](#ref-git-internals)]:

- `blob` — содержимое файла;
- `tree` — отображение `имя → (mode, тип, hash)` для одного каталога;
- `commit` — `tree` корневого снимка плюс ссылки на родительские коммиты, автор, сообщение;
- `tag` — подписанная ссылка на конкретный коммит.

Коммит фиксирует всё состояние репозитория одной строкой-хешем: изменение любого файла каскадирует вверх до commit hash. Исторически Git использовал SHA-1; с 2018 года ведётся миграция на SHA-256 (`git init --object-format=sha256`), мотивированная атакой SHAttered [[14](#ref-shattered)].

### Bitcoin

Каждый блок Bitcoin содержит в заголовке поле `merkle_root` — корень дерева над всеми транзакциями блока [[5](#ref-nakamoto)]. Это даёт два важных свойства:

- **Лёгкие клиенты (SPV).** SPV-кошелёк хранит только заголовки блоков (~80 байт каждый) и при подтверждении конкретной транзакции запрашивает у полного узла её audit path. По заголовку и audit path SPV восстанавливает merkle root и сравнивает с хранимым — без скачивания самого блока.
- **Эффективность proof'а.** Для блока с ~3000 транзакций audit path содержит ~12 хешей (~384 байта).

Bitcoin использует «дублирующее» правило для нечётного числа листьев — отсюда уязвимость **CVE-2012-2459** [[12](#ref-cve-2012-2459)], закрытая в Bitcoin Core явной проверкой невозможности дублей. Bloom-фильтрация транзакций для SPV описана в **BIP-37**, но позже была признана уязвимой к атакам по корреляции и в современных кошельках вытесняется compact block filters (BIP-157/158).

### Ethereum

Заголовок блока Ethereum содержит **три** корня деревьев Меркла [[6](#ref-yellowpaper), Appendix D]:

- `stateRoot` — состояние всех аккаунтов и storage-слотов;
- `transactionsRoot` — транзакции блока;
- `receiptsRoot` — receipts (события, gas used, статус) транзакций.

Все три используют **Merkle Patricia Trie** (MPT) — гибрид трая и дерева Меркла, оптимизированный для разреженного key-value-хранилища с криптографическими доказательствами. Краткое описание самого MPT см. в разделе [Варианты](#варианты).

### Cassandra и DynamoDB anti-entropy

При репликации поэлементное сравнение двух копий нереалистично: у каждой реплики могут быть гигабайты данных. Apache Cassandra и Amazon Dynamo строят над диапазонами ключей merkle tree и обмениваются корнями [[8](#ref-dynamo)] [[13](#ref-cassandra-repair)]:

1. координатор инициирует repair на диапазоне токенов;
2. каждая реплика строит дерево Меркла поверх своих SSTable'ов в этом диапазоне;
3. реплики обмениваются корнями; при совпадении ничего делать не нужно;
4. при расхождении рекурсивно сравниваются корни левого и правого поддеревьев — спуск идёт только в те ветви, где обнаружены различия.

Сложность по обмену — `O(diff · log n)`, где `diff` — число различающихся листьев, а не размер всего набора. Это даёт практичный anti-entropy даже для терабайтных таблиц.

### Certificate Transparency (RFC 6962)

CT — публичный append-only лог TLS-сертификатов; любой желающий может убедиться, что сертификат, выпущенный CA, попал в лог [[3](#ref-rfc-6962)] [[4](#ref-rfc-9162)]. Лог реализован как append-only дерево Меркла; центральный объект протокола — **Signed Tree Head (STH)**: подписанная пара `(размер дерева, корень)`, которую логи периодически публикуют.

Два класса доказательств:

- **Inclusion proof** — стандартный audit path для одного сертификата;
- **Consistency proof** — последовательность хешей, демонстрирующая, что новый STH получен **append-only**-расширением старого STH (никакая часть истории не была переписана).

Браузеры (Chrome, Safari) требуют, чтобы публично доверенные сертификаты включали **SCT** (Signed Certificate Timestamp) — обязательство лога опубликовать сертификат. RFC 9162 уточняет CT v2.0 с поддержкой нескольких хеш-функций и улучшенной структурой записей.

### IPFS и content-addressable storage

IPFS строит файловую систему как **Merkle DAG** [[9](#ref-ipfs)]. Каждый объект (файл, директория) идентифицируется **CID** (Content Identifier) — мультихешем своего содержимого. Большие файлы разбиваются на чанки, организованные в дерево; директории — в ноды-карты `имя → CID`. Доступ к контенту по CID гарантирует, что данные совпадают с теми, что были адресованы — манипуляция изменит CID. Сериализация — `dag-pb` (исторический формат) и `dag-cbor` (новый, на CBOR).

### ZFS, Btrfs, dm-verity

Файловые системы ZFS и Btrfs хранят хеши блоков в указателях родительских блоков, образуя дерево Меркла поверх структуры файла или метаданных — это даёт детектирование bit-rot при чтении (read-verify) и поддерживает целостный snapshot.

**dm-verity** в Linux (используется в Android verified boot, Chrome OS) располагает над read-only-разделом дерево Меркла поверх блоков фиксированного размера; ядро при чтении сверяет хеш блока со своим уровнем дерева вверх до подписанного root hash. Это обеспечивает целостность исполняемой системы при загрузке.

## Варианты

### Merkle Patricia Trie

В Ethereum используется **Merkle Patricia Trie** (MPT) — модификация бинарного дерева Меркла, заточенная под разреженные key-value-данные с переменной длиной ключей [[6](#ref-yellowpaper), Appendix D]. Узлы трёх типов:

- **leaf** — пара `(оставшийся nibble-путь, значение)`;
- **extension** — общая часть пути `(nibble-prefix, ссылка на следующий узел)` для сжатия цепочек;
- **branch** — массив из 17 ссылок (16 nibble-веток и опциональное значение в самой ноде).

Каждый узел сериализуется RLP-кодеком и хешируется Keccak-256; значение в родителе — это хеш дочернего узла. MPT даёт криптографические доказательства как наличия (`stateProof`), так и отсутствия (отсутствие пути в траe) — последнее особенно полезно для stateless-клиентов.

### Sparse Merkle Trees (SMT)

**Sparse Merkle Tree** [[10](#ref-smt)] — дерево фиксированной глубины (например, 256 уровней при ключах в 32 байта), в котором почти все листья пусты. Ключевые особенности:

- ключ адресует конкретный лист по своим битам — путь от корня к листу длины `H` уровней;
- пустые поддеревья имеют **предвычисленные** хеши (`empty_h`, где `h` — высота поддерева), что делает дерево разреженным по памяти при заполнении ленивого вида;
- **non-membership proof** доказывается тривиально: показывается, что лист по адресу-ключу содержит нулевое значение (или специальный маркер `nil`).

SMT используются в zk-rollup'ах (zkSync, Plasma Cash), системах накопления состояния (накопители в IBC) и решениях для privacy-preserving credentials.

### Verkle Trees

**Verkle Tree** — современное развитие идеи Меркла, заменяющее хеш-функцию на **векторное полиномиальное обязательство** (vector commitment). Вместо хеширования двух дочерних хешей вычисляется коммитмент к вектору **многих** дочерних хешей; благодаря свойствам полиномиальных обязательств (KZG или IPA) можно построить **multi-proof**, доказывающий включение группы листьев почти за `O(1)` по числу листьев.

Практический мотив для Ethereum — переход на **stateless-клиенты**: при ширине ветвления 256 размер witness падает с десятков мегабайт (при бинарном MPT с Keccak) до сотен килобайт. Ethereum Foundation описывает структуру в посте Ballet & Feist (2021) [[11](#ref-verkle-ef)]; академическое введение — Kuszmaul (2019) [[15](#ref-kuszmaul)]; внедрение запланировано как часть **Verge upgrade** через EIP-семейство 6800/4762 (на момент апреля 2026 — на тестнетах).

## Сравнение с альтернативами

| Метод | Размер «доказательства» одного элемента | Поддерживает inclusion-proof | Anti-entropy | Поддерживает append без полной пересборки |
|---|---|---|---|---|
| Один хеш всего набора `H(d_1 \|\| ... \|\| d_n)` | весь набор | нет | нет | нет |
| Список чексумм `[H(d_i)]` | `O(n)` бит | да (тривиально) | `O(n)` обменов | да |
| HMAC-цепочка `H_i = H(H_{i-1} \|\| d_i)` | весь префикс до `i` | только префиксное | нет | да |
| **Дерево Меркла** | `O(log n)` бит | да | `O(diff · log n)` | да, амортизированно `O(log n)` |
| Per-item подпись `Sig(d_i)` | константа + проверка подписи | да | нет (асимметрично дорогой verify) | да |

Дерево Меркла — единственный из перечисленных методов, дающий одновременно компактное доказательство, эффективный anti-entropy и append без перестройки.
