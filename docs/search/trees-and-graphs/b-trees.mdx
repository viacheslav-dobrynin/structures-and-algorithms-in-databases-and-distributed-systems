# B-деревья

## Мотивация

Классические структуры данных — бинарные деревья поиска, AVL-деревья, красно-чёрные деревья — спроектированы с расчётом на хранение данных в оперативной памяти. В этой модели стоимость операции измеряется числом сравнений ключей, а каждое обращение к узлу обходится одинаково дёшево.

В базах данных картина принципиально иная. Данные хранятся на диске, и доступ к диску на несколько порядков медленнее доступа к RAM: операция чтения с HDD занимает около 10 мс, что эквивалентно ~10 миллионам инструкций процессора. Даже SSD на несколько порядков медленнее оперативной памяти при случайном доступе.

Рассмотрим сбалансированное бинарное дерево (AVL) с миллионом записей: его высота составляет около 20 уровней. Поиск одного ключа потребует 20 чтений с диска — примерно 200 мс. Это неприемлемо медленно для СУБД, которой нужно обрабатывать тысячи запросов в секунду.

Решение — **уменьшить высоту дерева** за счёт увеличения количества ключей и потомков в каждом узле. Если узел хранит не 1, а 100 ключей (и имеет 101 потомка), высота дерева на том же миллионе записей составит всего `log_101(1000000) ≈ 3`. Три обращения к диску вместо двадцати — качественный скачок в производительности.

Именно эту идею реализует **B-дерево** (B-tree), предложенное Рудольфом Байером (Rudolf Bayer) и Эдвардом МакКрайтом (Edward McCreight) в 1972 году.

## Структура

### Определение и параметры

B-дерево степени `t` (минимальной степени, *minimum degree*) — это сбалансированное дерево поиска со следующими свойствами:

- Каждый узел содержит от `t - 1` до `2t - 1` ключей (кроме корня, который может содержать от 1 ключа).
- Каждый внутренний узел с `k` ключами имеет ровно `k + 1` потомков.
- Все листья находятся на одной глубине — дерево **идеально сбалансировано**.
- Ключи в узле упорядочены: `key_1 < key_2 < ... < key_k`.
- Ключи разбивают диапазоны: все ключи в `i`-м поддереве строго больше `key_{i-1}` и строго меньше `key_i`.

Параметр `t` выбирается так, чтобы узел соответствовал размеру дисковой страницы (обычно 4–16 КБ). Если запись занимает 100 байт, страница 4 КБ вместит около 40 записей, то есть `t ≈ 20`.

### Хранение записей

Каждый узел хранится как отдельная страница на диске. Типичная страница содержит:

- Массив ключей `[key_1, key_2, ..., key_k]`
- Массив указателей на потомков `[c_0, c_1, ..., c_k]` (для внутренних узлов)
- Ссылки на записи данных (для листовых узлов — в варианте B+-дерева)
- Счётчик текущего числа ключей `n`
- Флаг «является ли узел листовым»

### B+-дерево

На практике в СУБД почти всегда используется вариант **B+-дерево** (B+-tree):

- Все **ключи и указатели на данные** хранятся только в **листовых узлах**.
- **Внутренние узлы** хранят лишь ключи-разделители (копии ключей из листьев) и указатели на потомков — они служат только для навигации.
- Все **листовые узлы** связаны в **двусвязный список**, что позволяет эффективно выполнять диапазонные запросы: найдя начало диапазона, достаточно пройти по цепочке листьев.

Это ключевое преимущество перед оригинальным B-деревом: диапазонный запрос `WHERE age BETWEEN 20 AND 30` требует одного прохода по листовым страницам без возврата к внутренним узлам.

```
Внутренние узлы (навигация):
       [ 30 | 70 ]
      /     |     \
 [10|20] [40|50|60] [80|90]

Листовые узлы (данные, связный список):
→ [5|8] → [10|13|17|20] → [25|28|30] → [32|37|40] → ... → [93|97] →
```

## Операции

### Поиск (Search)

Поиск ключа `k` начинается с корня и выполняется за один проход сверху вниз:

1. В текущем узле найти наименьший индекс `i` такой, что `k ≤ key_i`.
2. Если `k = key_i`, ключ найден — вернуть указатель на запись.
3. Если узел листовой и равенства нет — ключ отсутствует.
4. Иначе рекурсивно выполнить поиск в `i`-м потомке.

```
Функция Search(узел N, ключ k):
    i = 0
    пока i < N.n и k > N.key[i]:
        i = i + 1
    если i < N.n и k == N.key[i]:
        вернуть N.data[i]   // ключ найден
    если N — лист:
        вернуть NULL         // ключ отсутствует
    вернуть Search(загрузить(N.child[i]), k)
```

На каждом уровне выполняется бинарный поиск по ключам узла — `O(log t)` сравнений. Итого: `O(t · log_t n)` сравнений, что при фиксированном `t` эквивалентно `O(log n)`.

### Вставка (Insert)

Вставка нового ключа всегда происходит в листовой узел. Ключевая проблема — переполнение: если узел уже содержит `2t - 1` ключей, он **разделяется** (split).

#### Алгоритм разделения узла (SplitChild)

При разделении полного дочернего узла `y` (содержащего `2t - 1` ключей):

1. Создать новый узел `z`.
2. Передать в `z` правую половину ключей узла `y`: ключи `[t, 2t-2]` (количество: `t - 1` ключей).
3. Средний ключ `y.key[t-1]` поднять в родительский узел.
4. Если `y` — внутренний, передать в `z` правую половину потомков: `[t, 2t-1]`.

```
Функция SplitChild(узел x, индекс i):
    // x — родитель, x.child[i] = y — полный узел (2t-1 ключей)
    y = x.child[i]
    z = новый узел
    z.n = t - 1
    для j = 0 до t-2:
        z.key[j] = y.key[j + t]
    если y не лист:
        для j = 0 до t-1:
            z.child[j] = y.child[j + t]
    y.n = t - 1
    // вставить z в список потомков x после y
    для j = x.n до i+1 (в обратном порядке):
        x.child[j+1] = x.child[j]
    x.child[i+1] = z
    // поднять средний ключ y в x
    для j = x.n-1 до i (в обратном порядке):
        x.key[j+1] = x.key[j]
    x.key[i] = y.key[t-1]
    x.n = x.n + 1
```

#### Вставка за один проход

Стандартная реализация выполняет вставку за **один проход сверху вниз**, превентивно разделяя все полные узлы на пути к листу:

1. Если корень полный, сначала разделить его (высота дерева увеличивается на 1).
2. Спускаясь к листу, при встрече полного дочернего узла немедленно разделить его.
3. Вставить ключ в листовой узел (который гарантированно не заполнен).

Это позволяет обойтись одним проходом без возврата вверх.

```
Функция Insert(дерево T, ключ k):
    r = T.root
    если r.n == 2t - 1:         // корень полный — разделить
        s = новый узел
        T.root = s
        s.child[0] = r
        SplitChild(s, 0)
        InsertNonFull(s, k)
    иначе:
        InsertNonFull(r, k)

Функция InsertNonFull(узел x, ключ k):
    i = x.n - 1
    если x — лист:
        сдвинуть ключи вправо, пока x.key[i] > k
        вставить k на освободившееся место
    иначе:
        пока i >= 0 и k < x.key[i]:
            i = i - 1
        i = i + 1
        если x.child[i] полный:
            SplitChild(x, i)
            если k > x.key[i]:
                i = i + 1
        InsertNonFull(x.child[i], k)
```

### Удаление (Delete)

Удаление сложнее вставки, так как необходимо поддерживать минимальное заполнение узла (`t - 1` ключей). Рассмотрим три случая.

**Случай 1 — ключ находится в листовом узле** с числом ключей `≥ t`:
Просто удалить ключ.

**Случай 2 — ключ находится во внутреннем узле**:

- 2a. Если левый потомок содержит `≥ t` ключей: заменить удаляемый ключ на **предшественника** (максимальный ключ левого поддерева) и рекурсивно удалить предшественника.
- 2b. Если правый потомок содержит `≥ t` ключей: аналогично с **преемником** (минимальный ключ правого поддерева).
- 2c. Если оба потомка содержат `t - 1` ключей: слить их, опустив разделяющий ключ вниз, и рекурсивно удалить ключ из получившегося узла.

**Случай 3 — ключ отсутствует во внутреннем узле, нужно спуститься в потомка** `c`:

- 3a. Если у `c` есть соседний брат с `≥ t` ключами: выполнить **ротацию** — переместить крайний ключ брата вверх в родителя, разделяющий ключ родителя спустить в `c`.
- 3b. Если все братья `c` содержат `t - 1` ключей: слить `c` с одним из братьев.

```
Функция Delete(узел x, ключ k):
    i = нижняя_граница(x.keys, k)
    если x — лист:
        если x.key[i] == k:
            удалить x.key[i]
        вернуть
    если i < x.n и x.key[i] == k:
        // ключ найден во внутреннем узле (случай 2)
        обработать случаи 2a, 2b, 2c
    иначе:
        // ключ ниже (случай 3)
        если x.child[i].n < t:
            обеспечить_наполнение(x, i)   // ротация или слияние
        Delete(x.child[i], k)
```

Если после удаления корень становится пустым (все ключи ушли вниз при слиянии), его единственный потомок становится новым корнем, и высота дерева уменьшается на 1.

## Сложность

| Операция        | Число обращений к диску | Время (CPU)   |
|-----------------|------------------------|---------------|
| Поиск           | `O(log_t n)`           | `O(t · log_t n)` |
| Вставка         | `O(log_t n)`           | `O(t · log_t n)` |
| Удаление        | `O(log_t n)`           | `O(t · log_t n)` |
| Диапазонный поиск | `O(log_t n + k/t)` | `O(t · log_t n + k)` |
| Пространство    | `O(n)`                 | —             |

Здесь `n` — число ключей в дереве, `t` — минимальная степень, `k` — число результатов в выборке.

**Высота дерева** не превышает `log_t((n+1)/2)`. При `t = 1000` и `n = 10^9` высота равна всего 3 — поиск требует трёх обращений к диску. Это и есть главное преимущество B-дерева перед бинарными деревьями.

**Диапазонный поиск в B+-дереве**: найдя начало диапазона за `O(log_t n)` операций, остаётся пройти по цепочке листьев. Одна страница хранит `≈ 2t - 1` ключей, поэтому для `k` результатов достаточно прочитать `⌈k / (2t-1)⌉` страниц.

## Применение в базах данных

### PostgreSQL

PostgreSQL использует B+-дерево как стандартный тип индекса (создаётся по умолчанию):

```sql
CREATE INDEX idx_users_email ON users (email);
-- Эквивалентно:
CREATE INDEX idx_users_email ON users USING btree (email);
```

B-tree индексы в PostgreSQL поддерживают операторы `=`, `<`, `<=`, `>`, `>=`, `BETWEEN`, `IN`, `IS NULL` и используются планировщиком для:

- **Index Scan** — обход дерева + случайный доступ к кучи (heap), эффективен при высокой селективности.
- **Index Only Scan** — если все нужные столбцы есть в индексе, обращение к куче не требуется.
- **Bitmap Index Scan** — при низкой селективности планировщик сначала строит битовую карту совпадений, затем читает страницы кучи последовательно.

Составной B-tree индекс `(a, b)` эффективен для условий на `a` или на `(a, b)`, но не только на `b`.

### MySQL (InnoDB)

InnoDB использует B+-дерево для всех индексов. **Кластерный индекс** (clustered index) — особенность InnoDB: таблица **физически хранится** в порядке первичного ключа. Листовые страницы кластерного индекса содержат не указатели на строки, а сами строки целиком.

```sql
-- Кластерный индекс — первичный ключ (или первый UNIQUE NOT NULL, или скрытый rowid)
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    amount DECIMAL,
    INDEX idx_customer (customer_id)   -- вторичный индекс
);
```

Вторичный индекс в InnoDB хранит значение первичного ключа вместо физического адреса строки. Это означает, что чтение по вторичному индексу всегда требует дополнительного поиска по кластерному индексу («двойной lookup»).

### SQLite

SQLite хранит каждую таблицу в отдельном B+-дереве. Страница по умолчанию — 4 КБ (настраивается до 65 КБ). При отсутствии первичного ключа SQLite создаёт скрытый 64-битный `rowid` как ключ кластерного B+-дерева.
