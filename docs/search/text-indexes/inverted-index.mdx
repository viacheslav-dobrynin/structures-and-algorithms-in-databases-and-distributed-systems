# Обратный индекс

## Мотивация

Задача полнотекстового поиска формулируется так: дана коллекция документов `D = {d_1, ..., d_N}`, в которой каждый документ представлен как последовательность токенов длины `L`; по поисковому запросу `q` нужно быстро найти документы, содержащие заданные слова (и желательно — отсортировать их по релевантности).

Наивное решение — **forward index** (прямой индекс, `doc → список термов`) плюс линейное сканирование. Для ответа на запрос нужно пройти каждый документ и проверить наличие термов, что даёт `O(N · L)` операций на запрос. Для веб-корпуса (`N ≈ 10^9`, `L ≈ 10^3`) это порядка `10^12` операций — неприемлемо для интерактивного поиска.

Ключевая идея — **инверсия**: для каждого терма заранее построить список документов, в которых он встречается. Тогда ответ на булев запрос `t_1 AND t_2` сводится к пересечению двух отсортированных списков, что выполняется за линейное время от суммы их длин, а не от общего размера коллекции.

Эту идею эксплуатируют практически все современные полнотекстовые поисковые движки:

- **Apache Lucene** — библиотека, лежащая в основе Elasticsearch, OpenSearch, Apache Solr [[3](#ref-lucene)];
- **Elasticsearch / OpenSearch** — распределённые поисковые системы над Lucene [[4](#ref-elasticsearch)];
- **PostgreSQL GIN** — встроенный тип индекса для полнотекстового поиска и массивов [[5](#ref-postgres-gin)];
- **Sphinx, Manticore, Tantivy** — независимые реализации инвертированного индекса.

Канонический обзор структур и алгоритмов для инвертированных файлов — survey Zobel & Moffat в ACM Computing Surveys [[11](#ref-zobel-moffat)].

## Структура обратного индекса

Обратный индекс состоит из двух основных компонентов: **словаря термов** (dictionary) и **постинг-листов** (postings lists).

### Словарь термов

Словарь отображает терм в указатель на его постинг-лист и хранит статистику — прежде всего **document frequency** `df(t)` (в скольких документах встречается терм). Типичные реализации:

- **Хеш-таблица** — `O(1)` на поиск, но не поддерживает префиксные и диапазонные запросы;
- **B-дерево / B+-дерево** — `O(log |V|)` на поиск, поддерживает сортированный обход и префиксные запросы;
- **Trie / префиксное дерево** — эффективно для префиксного поиска, но память больше;
- **FST (Finite State Transducer)** — компактное сжатое представление, используется в Lucene для словаря термов в сегменте (`.tim` / `.tip`) [[3](#ref-lucene)]. FST хранит отсортированный набор строк с общими префиксами и суффиксами, обеспечивая поиск за `O(|t|)`.

Размер словаря для англоязычного корпуса обычно подчиняется **закону Хипса**: `|V| ≈ k · N^β`, где `N` — общее число токенов, `β ≈ 0.4–0.6` [[1](#ref-manning-iir)].

### Постинг-лист

Постинг-лист — отсортированная по `docID` последовательность записей. Уровень детализации зависит от того, что нужно поддерживать:

- **Doc-only**: `[docID, docID, ...]` — достаточно для булева поиска;
- **Doc + tf**: `[(docID, tf), ...]` — нужно для ранжирования TF-IDF / BM25 (`tf` — частота терма в документе);
- **Doc + tf + positions**: `[(docID, tf, [pos_1, ..., pos_tf]), ...]` — нужно для фразового поиска, slop и wildcard.

### Пример

Пусть коллекция из трёх документов:

- `d_1 = "the quick brown fox"`;
- `d_2 = "the lazy brown dog"`;
- `d_3 = "quick brown dog jumps"`.

После токенизации и удаления стоп-слова `the` получается такой обратный индекс (с `tf` и позициями; позиции — 0-индексированные, стоп-слова сохраняют свои исходные позиции):

| Терм | df | Постинг-лист `(docID, tf, [positions])` |
|---|---|---|
| `brown` | 3 | `(1, 1, [2]), (2, 1, [2]), (3, 1, [1])` |
| `dog` | 2 | `(2, 1, [3]), (3, 1, [2])` |
| `fox` | 1 | `(1, 1, [3])` |
| `jumps` | 1 | `(3, 1, [3])` |
| `lazy` | 1 | `(2, 1, [1])` |
| `quick` | 2 | `(1, 1, [1]), (3, 1, [0])` |

Форматы Lucene дополнительно хранят смещения символов (character offsets) в исходном тексте (для подсветки) и полезную нагрузку (`payloads`) в файлах `.pos` / `.pay` [[3](#ref-lucene)].

### Схема: словарь + постинги

Логически словарь и постинг-листы образуют двухколонную структуру: слева — отсортированный набор термов с метаданными, справа — ссылки на постинг-листы:

| Словарь термов | → | Постинг-лист |
|---|:---:|---|
| `brown` \| df = 3 | → | `(1, 1, [2]) → (2, 1, [2]) → (3, 1, [1])` |
| `dog` \| df = 2 | → | `(2, 1, [3]) → (3, 1, [2])` |
| `quick` \| df = 2 | → | `(1, 1, [1]) → (3, 1, [0])` |

В реализации словарь обычно хранится в компактной структуре (FST в Lucene, B-дерево в GIN), а ссылка в правой колонке — это оффсет в файле `.doc` (Lucene) или указатель на posting list / posting tree (GIN).

## Построение индекса

### Пайплайн анализа текста

Перед индексированием текст проходит через цепочку преобразований:

1. **Токенизация** — разбиение на термы по границам слов. Для CJK-языков используется сегментация (например, Kuromoji для японского, SmartCN для китайского);
2. **Нормализация** — приведение к нижнему регистру, удаление диакритики (ASCII folding), обработка апострофов;
3. **Удаление стоп-слов** — `the`, `a`, `is`, `в`, `и`, `на` ... — частотные слова с низкой информационной ценностью. В современных системах (Lucene / Elasticsearch) стоп-слова по умолчанию **не удаляются** из индекса, потому что это ломает фразовый поиск и может снижать recall — удаление остаётся опциональным фильтром;
4. **Стемминг или лемматизация** — приведение словоформ к общей основе. Для английского — **Porter stemmer** [[8](#ref-porter)] и его развитие Snowball; для русского — **pymorphy2** и **Mystem**; для многих языков — Snowball-семейство.

Компромисс **recall / precision**: агрессивная нормализация (стемминг, удаление диакритики) повышает recall (больше совпадений), но снижает precision (больше шума). Лемматизация точнее стемминга, но дороже.

### Алгоритм SPIMI

**SPIMI** (Single-Pass In-Memory Indexing) — стандартный алгоритм построения индекса для коллекций, не помещающихся в память [[1](#ref-manning-iir)]:

```python
def spimi_invert(token_stream, block_size_limit):
    """
    token_stream — поток пар (term, docID).
    Возвращает список файлов-блоков с отсортированными частичными индексами.
    """
    block_id = 0
    dictionary = {}                      # term -> postings list
    current_size = 0
    block_files = []

    for term, doc_id in token_stream:
        if term not in dictionary:
            dictionary[term] = []        # новый постинг-лист
        postings = dictionary[term]
        if not postings or postings[-1] != doc_id:
            postings.append(doc_id)      # гарантирует отсортированность
            current_size += 1
        if current_size >= block_size_limit:
            block_files.append(
                write_sorted_block(dictionary, block_id)
            )
            dictionary = {}
            current_size = 0
            block_id += 1

    if dictionary:
        block_files.append(write_sorted_block(dictionary, block_id))
    return block_files                   # затем — merge всех блоков
```

После формирования блоков выполняется многопутевое слияние (merge): оно объединяет частичные индексы в один глобальный. Альтернатива — **BSBI** (Blocked Sort-Based Indexing): собирает `(termID, docID)`-пары, сортирует блоками и сливает; BSBI требует предварительного отображения `term → termID`, тогда как SPIMI работает напрямую с термами и естественно растит постинг-лист.

## Поиск

### Булев AND: пересечение двух отсортированных постингов

Если оба списка отсортированы по `docID`, пересечение ищется «двумя указателями» за `O(|p_1| + |p_2|)`:

```python
def intersect(p1, p2):
    """p1, p2 — отсортированные списки docID."""
    i, j = 0, 0
    result = []
    while i < len(p1) and j < len(p2):
        if p1[i] == p2[j]:
            result.append(p1[i])
            i += 1
            j += 1
        elif p1[i] < p2[j]:
            i += 1
        else:
            j += 1
    return result
```

Для запроса из `k` термов пересечение выгодно выполнять в порядке возрастания `df`: это быстрее всего сокращает промежуточный результат [[1](#ref-manning-iir)].

### Skip-pointers

Чтобы ускорить пересечение, в постинг-листы добавляют **skip-указатели**: через каждые `√|p|` элементов хранится ссылка на позицию дальше по списку. Когда `p_1[i] < p_2[j]`, можно «прыгнуть» по skip-указателю, если значение в точке прыжка всё ещё не превышает `p_2[j]`:

```python
def intersect_with_skips(p1, p2):
    i, j = 0, 0
    result = []
    while i < len(p1) and j < len(p2):
        if p1[i] == p2[j]:
            result.append(p1[i])
            i += 1
            j += 1
        elif p1[i] < p2[j]:
            # попытаться прыгнуть вперёд по skip-указателю
            if has_skip(p1, i) and p1[skip_target(p1, i)] <= p2[j]:
                while has_skip(p1, i) and p1[skip_target(p1, i)] <= p2[j]:
                    i = skip_target(p1, i)
            else:
                i += 1
        else:
            if has_skip(p2, j) and p2[skip_target(p2, j)] <= p1[i]:
                while has_skip(p2, j) and p2[skip_target(p2, j)] <= p1[i]:
                    j = skip_target(p2, j)
            else:
                j += 1
    return result
```

Шаг `√|p|` теоретически минимизирует ожидаемое число сравнений для равномерных распределений [[1](#ref-manning-iir)]. Lucene использует более сложную многоуровневую skip-структуру внутри блочных постингов [[3](#ref-lucene)].

### Фразовый поиск

Запрос `"brown dog"` требует, чтобы в документе эти два терма стояли подряд. Для каждого кандидата (документа в пересечении) нужно проверить позиции:

```python
def phrase_match(positions_t1, positions_t2, gap=1):
    """
    Проверяет, есть ли в документе пара позиций (p1, p2),
    такая что p2 - p1 == gap. positions_t1, positions_t2 отсортированы.
    """
    i, j = 0, 0
    while i < len(positions_t1) and j < len(positions_t2):
        diff = positions_t2[j] - positions_t1[i]
        if diff == gap:
            return True
        elif diff < gap:
            j += 1
        else:
            i += 1
    return False
```

Для фразы из `k` слов применяется каскадная проверка пар `(t_i, t_{i+1})` с соответствующими `gap = 1`. Альтернатива — **biword index** (индексируются пары соседних слов, ответ на 2-словные фразы за один lookup) или **k-gram index**; оба выигрывают в скорости и проигрывают в размере.

### Wildcard и префиксный поиск

- **Префиксный запрос** `comput*` выполняется обходом отсортированного словаря (в B-дереве или FST) от `comput` до следующей по лексикографии строки. Lucene хранит префиксные границы в FST, что даёт `O(|prefix|)`;
- **Wildcard с `*` внутри** — через **k-gram index** (индекс по биграммам / триграммам термов) или **permuterm index** (все ротации терма с маркером `$`);
- **Fuzzy search** — Lucene использует Levenshtein automaton в комбинации с FST.

## Ранжирование

Для большинства запросов пользователю нужен не просто список документов, а отсортированный по релевантности. Классические формулы ранжирования — **TF-IDF** и **BM25**.

### TF-IDF

Идея: релевантность растёт с частотой терма в документе (`tf`) и с редкостью терма в коллекции (`idf`). Классический вариант [[1](#ref-manning-iir)]:

```
tf-idf(t, d) = tf(t, d) · log(N / df(t))
```

где:
- `tf(t, d)` — частота терма `t` в документе `d` (встречаются варианты: сырая, логарифмическая `1 + log(tf)`, нормализованная на максимум);
- `N` — общее число документов;
- `df(t)` — число документов, содержащих `t`.

Оценка релевантности документа запросу — сумма `tf-idf(t, d)` по всем термам запроса.

### BM25

**BM25** (Best Matching 25) — вероятностная модель ранжирования Robertson–Spärck Jones, эмпирически превосходящая TF-IDF на большинстве бенчмарков [[2](#ref-robertson-bm25)]. Базовая формула:

```
                         tf(t, d) · (k1 + 1)
BM25(t, d) = IDF(t) · ───────────────────────────────────────
                      tf(t, d) + k1 · (1 − b + b · |d| / avgdl)

IDF(t) = log( (N − df(t) + 0.5) / (df(t) + 0.5) + 1 )
```

Параметры:
- `k1` — контролирует скорость насыщения по `tf`; типичные значения `k1 ∈ [1.2, 2.0]`, по умолчанию в Lucene `k1 = 1.2`;
- `b` — контролирует нормализацию по длине документа; `b ∈ [0, 1]`, по умолчанию `b = 0.75`;
- `avgdl` — средняя длина документа в коллекции;
- IDF в приведённой форме (`+1` внутри `log`) — вариант Lucene `BM25Similarity`, гарантирующий неотрицательность [[3](#ref-lucene)]. В оригинале Robertson–Zaragoza использовалась форма `log((N − df + 0.5) / (df + 0.5))`, которая при `df > N/2` может стать отрицательной [[2](#ref-robertson-bm25)].

### Насыщение по TF

Главная особенность BM25 — **насыщение**: вклад `tf` в оценку ограничен сверху `(k1 + 1)`, тогда как в чистой TF-IDF он растёт линейно. На практике это означает: «документ, в котором слово встречается 100 раз, не в 100 раз релевантнее документа, где оно встречается один раз».

### Learning-to-rank и нейронные reranker-ы

BM25 — линейная формула с ~2 гиперпараметрами, и её потолок достигается быстро. Поверх инвертированного индекса обычно строят многоступенчатое ранжирование: первая стадия (retrieval) оставляет top-`K` кандидатов по BM25, вторая — переранжирует их моделью **learning-to-rank** (LambdaMART, XGBoost, нейронные модели) или **cross-encoder**-ом (BERT-подобные модели, напрямую сравнивающие запрос и документ). Retrieval-стадию дополняют **плотным** поиском по [HNSW](../vector-indexes-and-quantization/hnsw.mdx) и комбинируют результаты через **Reciprocal Rank Fusion** или обучаемые смешивающие модели.

## Компрессия постинг-листов

Постинг-листы — основная часть объёма индекса. Сжатие критично и для размера на диске, и для пропускной способности кеша: чем меньше байт, тем больше документов помещается в L2/L3.

### Gap-кодирование (delta encoding)

`docID` в постинге монотонно возрастают, поэтому вместо абсолютных значений хранят **разности (gaps)**. Пример:

```
docIDs:  [12, 20, 41, 42, 100]
gaps:    [12,  8, 21,  1,  58]
```

Gaps — в среднем маленькие числа (особенно для частых термов, где плотность постингов высока), и их можно кодировать плотнее, чем 32 бита на `docID`.

### Variable-Byte (VByte)

VByte кодирует неотрицательное целое в последовательность байт по 7 бит данных + 1 бит-продолжение:

- `0xxxxxxx` — последний байт числа (continuation = 0);
- `1xxxxxxx` — промежуточный байт (continuation = 1).

```python
def vbyte_encode(n):
    """Кодирует неотрицательное целое в байты VByte (little-endian 7-бит)."""
    assert n >= 0
    bytes_out = []
    while True:
        b = n & 0x7F       # младшие 7 бит
        n >>= 7
        if n == 0:
            bytes_out.append(b)    # последний байт: старший бит = 0
            break
        bytes_out.append(b | 0x80) # промежуточный: старший бит = 1
    return bytes_out

def vbyte_decode(stream):
    n = 0
    shift = 0
    for b in stream:
        n |= (b & 0x7F) << shift
        if (b & 0x80) == 0:
            yield n
            n = 0
            shift = 0
        else:
            shift += 7
```

Пример для `n = 130`: в двоичной системе `130 = 0b1000_0010`. Младшие 7 бит — `0b0000010 = 2`, после сдвига на 7 бит остаётся `1`. Первый байт — `0b1000_0010 = 0x82` (данные `2`, continuation = 1), второй — `0b0000_0001 = 0x01` (данные `1`, continuation = 0). Итого 2 байта вместо 4 для `int32`.

VByte прост, но имеет ограничение: каждое число занимает ≥ 1 байт, даже если gap равен 1.

### Gamma, Delta, Golomb-Rice

Битовые кодировки, которые выигрывают VByte для очень маленьких значений [[7](#ref-witten-mg)]:

- **Gamma-код** представляет `n` как `⌊log_2 n⌋` нулей + унарный ограничитель + `⌊log_2 n⌋` младших бит; длина кодового слова ≈ `2 · log_2 n` бит;
- **Delta-код** — то же, но префикс (длина) кодируется gamma-кодом; выигрышнее gamma для больших `n`;
- **Golomb-Rice** — параметрический код, оптимальный для геометрического распределения; применим, если известна средняя плотность (например, `1 / avg_gap`).

### PForDelta, FOR, блочное сжатие

Современные движки сжимают постинги **блоками** фиксированного размера (Lucene — 128 значений):

- **FOR** (Frame of Reference) — в блоке все значения кодируются фиксированным числом бит, равным `⌈log_2 max⌉`;
- **PForDelta** [[10](#ref-zukowski-pfor)] — то же, но несколько «выбросов» с большим значением кодируются отдельно (exceptions), а остальные — фиксированной шириной. Даёт отличное сжатие на коротких gap-ах; для быстрого декодирования применяют SIMD-векторизацию [[9](#ref-lemire-pfd)];
- Lucene с 9.x использует PFOR-подобный формат (`Lucene90PostingsFormat` и более новые ревизии) для файлов `.doc` и `.pos` [[3](#ref-lucene)].

Компромисс — между плотностью кодирования и скоростью декодирования. В инвертированных индексах пересечение постингов выполняется миллиарды раз, поэтому приоритет — скорость, а не максимальное сжатие: отсюда популярность блочных FOR / PForDelta вместо более плотных энтропийных кодов.

## Обновления: сегменты и merge

Обратный индекс плохо обновляется «на месте»: изменение одного постинг-листа задевает всё сжатие блока. Решение, используемое Lucene и всеми движками над ним — **иммутабельные сегменты** [[3](#ref-lucene)]:

1. Новые документы попадают в **in-memory buffer** и индексируются там;
2. При достижении порога (размер буфера, таймаут) буфер **сбрасывается** (flush) в новый сегмент на диске — маленький, но полностью самодостаточный индекс;
3. Удаление документа не переписывает сегмент, а помечается в битмапе `liveDocs` (**tombstone**);
4. Периодически **merge policy** (в Lucene по умолчанию **TieredMergePolicy**) выбирает несколько сегментов похожего размера и сливает их в один, попутно физически удаляя tombstone-записи.

Это сильно напоминает **LSM-tree** (log-structured merge tree): та же идея — оптимизировать запись за счёт накопления в памяти и периодических merge.

Trade-offs:

- **Частый flush** — низкая задержка индексации, много мелких сегментов, медленнее поиск (больше сегментов — больше seek);
- **Редкий flush** — быстрее поиск, но дольше буфер в памяти; опасность потерять данные при сбое (защищается **translog** / **WAL**);
- **Агрессивный merge** — стабильно мало сегментов, но пики I/O и CPU;
- **Мягкий merge** — низкое фоновое давление, но индекс может деградировать (много сегментов, много tombstones).

## Обратный индекс vs forward index

| Критерий | Обратный индекс | Forward index (doc-values, stored fields) |
|---|---|---|
| Основа | `term → postings` | `doc → значения поля` |
| Полнотекстовый поиск | Быстро (`O(\|postings\|)`) | Медленно (`O(N · L)`) |
| Сортировка / агрегация по полю | Медленно | Быстро (`O(\|docs\|)`) |
| Фасетный поиск, aggregations | Плохо | Хорошо |
| Вывод полей документа в ответе | Нет | Да (stored fields) |
| Обновление | Сегментами + merge | Сегментами + merge |

Lucene хранит **обе** структуры параллельно: инвертированные постинги (`.tim`, `.doc`, `.pos`) для поиска и **doc values** (`.dvd`, `.dvm`) для сортировки и агрегаций, плюс **stored fields** (`.fdt`, `.fdx`) для возврата документа в ответе [[3](#ref-lucene)].

## Практическое применение

### Apache Lucene

Lucene — эталонная реализация инвертированного индекса. Ключевые файлы сегмента (Lucene 9.x / 10.x, формат `Lucene99Codec` и более новые ревизии) [[3](#ref-lucene)]:

- `.tim` / `.tip` — словарь термов (FST-индекс в `.tip` для блочного поиска, основной контент в `.tim`);
- `.doc` — постинг-листы `docID` и `tf`, сжатые блочным PFOR-подобным кодом;
- `.pos` — позиции термов;
- `.pay` — полезная нагрузка (payloads) и смещения символов (character offsets) для подсветки;
- `.fdt` / `.fdx` — stored fields и индекс быстрого доступа к ним;
- `.dvd` / `.dvm` — doc values (колоночные данные для сортировки и агрегаций);
- `.nvd` / `.nvm` — norms (длины полей для BM25).

**Codec architecture** — способ абстрагировать форматы файлов: каждая версия Lucene выпускает свой `Codec` (`Lucene90PostingsFormat`, `Lucene99Codec` и т. д.), и индекс может содержать сегменты разных версий. Это позволяет эволюционировать формат без переиндексации всего корпуса.

### Elasticsearch и OpenSearch

Elasticsearch и форк OpenSearch строятся поверх Lucene: каждый **шард** — это полноценный Lucene-индекс. Над Lucene добавлены:

- REST API и JSON-DSL запросов (`query`, `bool`, `match_phrase`, ...);
- Распределённое шардирование и реплики;
- Анализаторы: встроенные (`standard`, `russian`, `english`, `keyword`, `whitespace`) и настраиваемые цепочки (char filters → tokenizer → token filters), например:

```json
{
  "analysis": {
    "analyzer": {
      "ru_text": {
        "type": "custom",
        "tokenizer": "standard",
        "filter": ["lowercase", "russian_morphology", "ru_stopwords"]
      }
    }
  }
}
```

При смене анализатора требуется **полная переиндексация** — обратный индекс существующих сегментов построен по старым правилам токенизации.

### PostgreSQL GIN

**GIN** (Generalized Inverted Index) — встроенный в PostgreSQL инвертированный индекс, пригодный для полнотекстового поиска (`tsvector`), массивов, JSONB, расширения `pg_trgm` [[5](#ref-postgres-gin)] [[6](#ref-pgtrgm)].

Внутреннее устройство [[5](#ref-postgres-gin)]:

- **Entry tree** — B-дерево по ключам (термам). Каждый листовой элемент указывает на постинг-лист;
- **Posting list / posting tree** — если в списке мало TID-ов, они хранятся прямо в записи entry tree (posting list внутри строки). Когда список разрастается — перемещается в отдельный **posting tree** (B-дерево по `(block, offset)` TID-ам);
- **Pending list** — дополнительная однонаправленная очередь новых записей. Режим `fastupdate=on` (по умолчанию включён) ускоряет вставку: новые постинги складываются в pending list без немедленного обновления основной структуры. Периодически `autovacuum` или явный `VACUUM` перемещает их в основной индекс.

Пример создания полнотекстового индекса и запроса:

```sql
-- Индекс по морфологически нормализованному тексту
CREATE INDEX articles_body_fts
  ON articles
  USING GIN (to_tsvector('russian', body));

-- Запрос: документы, релевантные 'обратный индекс'
SELECT id, title,
       ts_rank(to_tsvector('russian', body),
               plainto_tsquery('russian', 'обратный индекс')) AS rank
FROM articles
WHERE to_tsvector('russian', body) @@ plainto_tsquery('russian', 'обратный индекс')
ORDER BY rank DESC
LIMIT 10;
```

Оператор `@@` возвращает `true`, если `tsvector` соответствует `tsquery`. `ts_rank` — встроенная функция ранжирования (на основе TF и близости термов; BM25 как таковой в Postgres не встроен, но существует в расширениях).

**pg_trgm** [[6](#ref-pgtrgm)] даёт индексы по триграммам строк и поддерживает `ILIKE '%...%'`, операторы подобия `%` и `<->` (distance), делая возможным fuzzy search и wildcard на уровне GIN/GiST.

## Ограничения и компромиссы

- **Асимметрия индексации и поиска.** Построение индекса дорого (часы для больших корпусов), поиск — дёшев. Эту асимметрию бесполезно пытаться обратить;
- **Смена анализатора = переиндексация.** Любое изменение токенизации, стемминга или стоп-листа требует перестроить весь индекс. Отсюда — осторожность с экспериментами над анализаторами в продакшене;
- **Нет семантики.** Обратный индекс сопоставляет поверхностные формы (или их основы). Синонимы, парафразы, кросс-язычность решаются либо словарями синонимов (дорого и хрупко), либо плотным поиском по эмбеддингам — см. [HNSW](../vector-indexes-and-quantization/hnsw.mdx);
- **Гибридный поиск.** На практике лучшие результаты даёт комбинация: первая стадия — BM25-retrieval по инвертированному индексу, вторая — dense retrieval по HNSW; результаты объединяются через **Reciprocal Rank Fusion** или обучаемую комбинацию скоров;
- **Не подходит для аналитики и сортировок** — для них рядом держат doc values / column store;
- **Большие словари.** В корпусах с большим числом редких токенов (код, логи, URL) словарь распухает; помогает агрессивная нормализация и ограничение длины термов.
